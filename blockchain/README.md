# blockchain/

This directory contains OpenRealm's orchestration-layer implementation.

The blockchain/orchestration layer is intentionally separate from the real-time runtime simulation. It handles low-frequency, ownership/economy-oriented operations while the C/C++ runtime layer continues to own moment-to-moment world behavior.

## Scope

The current implementation covers:

- player registration / unregistration
- wallet-linked identity handles
- expiring runtime-session authorization for gameplay/runtime signers
- globally readable orchestration parameters through `GlobalParams`
- NFT-backed chunk ownership claims
- minimum-priced free chunk purchases with refundable abandonment back to the free pool
- owner-managed delegated editors
- runtime-facing chunk permission/state queries
- fixed-price marketplace listings
- English auctions
- unified sale-state queries for runtime/UI reads
- marketplace fee capture
- artifact generation and deployment scripting

## Non-goals

This layer does **not** own:

- frame-by-frame gameplay simulation
- voxel editing propagation
- player movement
- physics
- chunk meshing/rendering
- peer discovery or routing

## Contracts

### `GlobalParams`
Responsibilities:
- expose shared blockchain/runtime tuning values from one contract
- hold `MIN_CHUNK_COORD`, `MAX_CHUNK_COORD`, `MIN_CHUNK_PRICE`, `MAX_FEE_BPS`, and `MIN_AUCTION_DURATION`
- give runtime/UI code a single address to query for chain-level marketplace/chunk parameters

### `PlayerRegistry`
Responsibilities:
- Register/Unregister one active player identity per wallet
- enforce unique active handles
- store a metadata URI per registered player
- support handle updates and metadata updates
- preserve a stable `playerId` across Unregister/re-Register cycles
- authorize expiring runtime-session keys for runtime/off-chain message signing
- resolve a signer address into a direct registered wallet account or an authorized runtime session owner
- emit handle-hash-friendly events for indexers

Primary runtime-facing reads:
- `PlayerIdOf(account)`
- `ResolveRuntimeAccount(actor)`
- `GetRuntimeSession(session)`

### `ChunkClaims`
Responsibilities:
- enforce chunk coordinate bounds through `GlobalParams`
- allow only registered players to claim free chunks
- require the exact `MIN_CHUNK_PRICE` payment when claiming a free chunk
- mint one NFT-backed ownership token per claimed chunk
- expose `TokenIdOfChunk(x, y)` for off-chain indexing and marketplace integration
- support voluntary abandonment with refund of the locked minimum chunk purchase price, plus direct owner transfer
- support delegated editor permissions that reset on ownership transfer
- expose an official marketplace transfer hook
- expose one-call runtime-facing permission/state reads for the next runtime layer

Primary runtime-facing reads:
- `CanEdit(x, y, account)`
- `EditorEpochOfChunk(x, y)`
- `GetChunkRuntimeState(x, y, actor)`
- `GetClaim(x, y)` / `GetClaimByTokenId(tokenId)`

`GetChunkRuntimeState(...)` is intended to be the main runtime integration surface. It returns:
- claim presence
- token id and coordinates
- owner address and stable `ownerPlayerId`
- claim timestamp
- editor/permission epoch for cache invalidation
- resolved actor account for the provided signer
- whether that actor can edit
- whether the signer is being used through an authorized runtime session

### `Marketplace`
Responsibilities:
- list claimed chunks for fixed-price sale at or above `MIN_CHUNK_PRICE`
- run English auctions with reserve price + minimum bid increment, with reserve price at or above `MIN_CHUNK_PRICE`
- prevent simultaneous active sale modes on the same chunk token
- invalidate stale listings/auctions if ownership changes outside the marketplace
- transfer chunk ownership through `ChunkClaims`
- retain a configurable protocol fee in basis points, validated against `GlobalParams.MAX_FEE_BPS`
- allow the contract owner to withdraw accumulated fees
- emit chunk-key/token-id-oriented events for indexing
- expose unified sale-state reads for runtime/UI code

Primary runtime-facing reads:
- `GetSaleStateForChunk(x, y)`
- `GetSaleStateForToken(tokenId)`

These unified sale-state views expose whether a chunk currently has:
- no active sale
- an active fixed-price listing
- an active auction

along with the relevant seller, current pricing, bid, and freshness fields.

## Tooling

The local toolchain is intentionally lightweight:

- `solc` for compilation
- `ganache` for ephemeral local EVM execution in tests
- `ethers` for deployment and interaction
- `mocha` for the test runner

## Build, test, deploy

All commands below are run from `blockchain/`.

### 1. Install dependencies

```bash
npm install
```

Windows shortcut:

```bat
install-deps.bat
```

### 2. Build artifacts

```bash
npm run build
```

Windows shortcut:

```bat
build-contracts.bat
```

This compiles every Solidity source and writes JSON artifacts under `artifacts/`.

### 3. Run tests

```bash
npm test
```

Windows shortcut:

```bat
test-blockchain.bat
```

Expected result today:
- the orchestration-layer test suite passes
- Ganache may print a `µWS` compatibility warning on this machine and fall back to a pure Node.js implementation; that is noisy but non-blocking for local tests

### 4. Deploy to a local Ganache node

Start a local JSON-RPC chain in another shell, for example:

```bash
npx ganache --wallet.totalAccounts 10 --chain.chainId 31337
```

Windows shortcut:

```bat
start-ganache-local.bat
```

Then deploy from the test realm wrapper:

```bash
node ../realms/test/deploy.js --private-key YOUR_LOCAL_PRIVATE_KEY --owner YOUR_OWNER_ADDRESS
```

Windows shortcut:

```bat
..\realms\test\deploy-local.bat [privateKey] [ownerAddress]
```

If you omit both arguments, `realms\test\deploy-local.bat` uses the first account from the default local Ganache mnemonic.

Notes:
- `--owner` is optional; if omitted, the deployer wallet becomes the contract owner
- the test-realm wrapper reads `../realms/test/realm.json` for the realm name + RPC URL and writes a deployment record to `deployments/test.json`
- `GlobalParams` is deployed first and its address + parameter values are included in that deployment record for later runtime fetching

### 5. Deploy to another EVM-compatible network

Set environment variables or pass CLI flags:

```bash
RPC_URL=https://your-rpc.example \
PRIVATE_KEY=0xyourprivatekey \
OWNER_ADDRESS=0xyourowner \
MIN_CHUNK_PRICE_WEI=10000000000000000 \
FEE_BPS=500 \
npm run deploy -- --network sepolia
```

Equivalent explicit flags:

```bash
npm run deploy -- \
  --rpc https://your-rpc.example \
  --private-key 0xyourprivatekey \
  --owner 0xyourowner \
  --min-chunk-price-wei 10000000000000000 \
  --fee-bps 500 \
  --network sepolia
```

Deployment output:
- deployer/owner summary in stdout
- deployed contract addresses in stdout
- `deployments/<network>.json` written locally

### Quick local verification

If you just want the normal beginner loop for local work:

```bat
verify-local.bat
```

That runs:
- `npm run build`
- `npm test`

## Main tested flows

The current test suite covers:

1. player registration, handle uniqueness, and handle release on Unregister
2. runtime-session authorization and runtime-facing chunk permission resolution
3. paid NFT-backed free chunk claims and registered-only ownership transfer behavior
4. abandonment refunds that return chunks to the free pool at `MIN_CHUNK_PRICE`
5. fixed-price marketplace purchase flow, fee retention, and sale-state queries
6. English auction bidding, sale-state reads, and settlement
7. stale listing invalidation after direct token transfer
8. rejection of unregistered buyers

## Integration expectations for the next runtime layer

The upcoming runtime layer should treat this workspace as the source of **ownership/identity/market facts**, not real-time gameplay authority.

Practical runtime reads now available:
- resolve an incoming signer/session key through `PlayerRegistry.ResolveRuntimeAccount(actor)`
- read one-call chunk permission state through `ChunkClaims.GetChunkRuntimeState(x, y, actor)`
- cheaply invalidate permission caches through `ChunkClaims.EditorEpochOfChunk(x, y)`
- inspect sale state through `Marketplace.GetSaleStateForChunk(x, y)`

Recommended runtime usage pattern:
1. runtime receives an action signed by wallet or runtime-session key
2. runtime resolves the signer to the backing wallet/player account
3. runtime reads chunk runtime state for the targeted chunk
4. runtime enforces ownership/delegated-editor permissions off-chain
5. runtime optionally checks sale state for UX/business logic, not simulation authority

The runtime should not wait on chain finality for normal gameplay. On-chain state should inform authorization and ownership, not real-time simulation.

## Known deferred items

Still intentionally out of scope for this implementation:

- wallet rotation / recovery flows
- governance / upgrade patterns
- NFT metadata hosting / rich tokenURI generation
- royalty standards / third-party marketplace integration
- indexer/subgraph project files (events are prepared for them, but the indexer itself is not included yet)
- runtime message envelope standards beyond the on-chain runtime-session authorization primitive
